---
title: Documentation Versioning
description: How the Zarf documentation site serves multiple versions and how to cut a new one.
---

The documentation site uses the [`starlight-versions`](https://starlight-versions.vercel.app/)
plugin. The version currently being edited lives in `site/src/content/docs/` and is served as
**Latest**. Released versions are archived in `site/src/content/docs/<slug>/` and are reachable
through the version switcher in the site header.

## Cutting a new version

When a release is cut, archive the current docs as that version:

1. Add the new slug to the `versions` list in `site/astro.config.ts`, newest first:

   ```ts
   starlightVersions({
     versions: [{ slug: "v0.78" }, { slug: "v0.77" }],
   }),
   ```

2. Start the dev server once. The plugin snapshots the current docs, assets, and sidebar into
   `site/src/content/docs/v0.78/` (and matching `site/src/content/versions/v0.78.json`):

   ```sh
   cd site && npm run dev
   ```

3. Stop the server and commit the generated files. Continue editing `Latest` as usual.

## Authoring constraints

Archived pages are processed as MDX, so all documentation — including the generated CLI command
reference under `commands/` — must be valid MDX. In practice that means:

- No HTML comments (`<!-- ... -->`) or indented code blocks; use fenced code blocks instead.
- No bare `<...>` or `{...}` in prose; wrap them in backticks.
- Reference files outside `site/src/` with the `@examples` / `@packages` aliases
  (see `site/tsconfig.json`) rather than deep `../../` paths, so imports keep resolving when a
  page is copied into a version subdirectory.

The CLI command docs are generated by `make docs-and-schema`; that generator already enforces
these rules.
