---
title: "Variables to Values"
---

This guide explains how to migrate your package from variables to [package values](/ref/package-values). Migrating has several benefits:

- Intuitive map structure - especially beneficial for large yaml block configuration.
- Standard go-template engine with built-in and custom functions.
- Built-in objects such as Package and State.

## Verify

To verify that this change made no impact on your package run `zarf dev inspect manifests` before and after your migration. There should be no change aside from the `# Source` comment.

## Migrating Manifests and Files

Below shows an example migration for a regular text file. The below example shows manifests, however this format is applicable to files and Helm values files when templating is enabled.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: DATABASE_USERNAME
    default: "postgres"
```

```yaml
# config.yaml
data:
  username: ###ZARF_VAR_DATABASE_USERNAME###
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    manifests:
      - name: configmap
        template: true  # Required for Go templates
        files:
          - config.yaml
```

```yaml
# package-values.yaml
database:
  username: "postgres"
```

```yaml
# config.yaml
data:
  username: {{ .Values.database.username }}
```

### Structured Values

Legacy variables are strings, so injecting a block like `tolerations` meant storing indented YAML in a single variable and relying on `.variables.autoIndent`. Because values are a map, keep the structure native and render it with `toYaml`:

```yaml
# package-values.yaml
tolerations:
  - key: "dedicated"
    operator: "Exists"
    effect: "NoSchedule"
```

```yaml
# deployment.yaml
spec:
  template:
    spec:
      tolerations:
        {{- toYaml .Values.tolerations | nindent 8 }}
```

## Migrating Helm Charts

Helm charts don't use Zarf templating. Instead of substituting variables into a chart's values file, map Package Values directly onto the chart's values with `sourcePath` and `targetPath`.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: REPLICA_COUNT
    default: "3"

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        valuesFiles:
          - chart-values.yaml
```

```yaml
# chart-values.yaml
replicaCount: ###ZARF_VAR_REPLICA_COUNT###
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        values:
          - sourcePath: ".app.replicas"
            targetPath: ".replicaCount"
```

```yaml
# package-values.yaml
app:
  replicas: 3
```

`sourcePath` references your Package Values (everything under `.Values.*`) and `targetPath` references the chart's `values.yaml` structure.

### Templated Values Files

When [Zarf templating](/ref/package-values/#templating) built-ins are required use `templatedValuesFiles`. Migration then looks the same as [Manifests and Files](#migrating-manifests-and-files) and the values files has access to template objects such as `.State`.

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        templatedValuesFiles:
          - chart-values.yaml
```

```yaml
# chart-values.yaml
replicaCount: {{ .Values.app.replicas }}
imageRegistry: {{ .State.Registry.Address }}
```

## Migrating actions

Legacy variables are substituted into action `cmd` fields automatically. With values, actions require `template: true` to opt in, then reference values as Go templates. Actions that captured output into a variable with `setVariables` use `setValues` instead.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: DATABASE_USERNAME
    default: "postgres"

components:
  - name: app
    actions:
      onDeploy:
        after:
          - cmd: echo "user is ###ZARF_VAR_DATABASE_USERNAME###"
          - cmd: echo "generated-password-123"
            setVariables:
              - name: DATABASE_PASSWORD
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    actions:
      onDeploy:
        after:
          - cmd: echo "user is {{ .Values.database.username }}"
            template: true  # Required for Go templates
          - cmd: echo "generated-password-123"
            setValues:
              - key: .database.password
```

```yaml
# package-values.yaml
database:
  username: "postgres"
```
