---
title: "Variables to Values"
---

This guide explains how to migrate your package from variables to values. Migrating has several benefits:

- Values are have a map structure that is much more intuitive when declaring a large yaml block.
- Standard go-template engine with built-in and custom functions.
- Built-in objects such as Package and State
- Ability to Toggle on or off


## Verify

To verify that this change made no impact on your package run `zarf dev inspect manifests` before and after your migration. There should be no change aside from the `# Source` comment.

## Migrating manifests and files

Below shows an example migration for a regular text file. The below example shows manifests, however this format is applicable to files and Helm values files when templating is enabled.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: DATABASE_USERNAME
    default: "postgres"
```

```yaml
# config.yaml
data:
  username: ###ZARF_VAR_DATABASE_USERNAME###
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    manifests:
      - name: configmap
        template: true  # Required for Go templates
        files:
          - config.yaml
```

```yaml
# package-values.yaml
database:
  username: "postgres"
```

```yaml
# config.yaml
data:
  username: {{ .Values.database.username }}
```

## Migrating Helm charts

Helm charts don't use Zarf templating. Instead of substituting variables into a chart's values file, map Package Values directly onto the chart's values with `sourcePath` and `targetPath`.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: REPLICA_COUNT
    default: "3"

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        valuesFiles:
          - chart-values.yaml
```

```yaml
# chart-values.yaml
replicaCount: ###ZARF_VAR_REPLICA_COUNT###
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        values:
          - sourcePath: ".app.replicas"
            targetPath: ".replicaCount"
```

```yaml
# package-values.yaml
app:
  replicas: 3
```

`sourcePath` references your Package Values (everything under `.Values.*`) and `targetPath` references the chart's `values.yaml` structure.

### Keeping a templated values file

If you need templating logic or built-in objects that `sourcePath`/`targetPath` mappings can't express, move the file to `templatedValuesFiles` and replace the `###ZARF_VAR_###` substitutions with Go templates. Templated values files have Go templates rendered at deploy time, giving them access to the built-in [templating](/ref/package-values/#templating) functions and objects such as `.State`.

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    charts:
      - name: my-chart
        namespace: my-app
        localPath: charts/my-chart
        templatedValuesFiles:
          - chart-values.yaml
```

```yaml
# chart-values.yaml
replicaCount: {{ .Values.app.replicas }}
imageRegistry: {{ .State.Registry.Address }}
```

## Migrating actions

Legacy variables are substituted into action `cmd` fields automatically. With values, actions require `template: true` to opt in, then reference values as Go templates. Actions that captured output into a variable with `setVariables` use `setValues` instead.

### Before (Legacy)

```yaml
# zarf.yaml
variables:
  - name: DATABASE_USERNAME
    default: "postgres"

components:
  - name: app
    actions:
      onDeploy:
        after:
          - cmd: echo "user is ###ZARF_VAR_DATABASE_USERNAME###"
          - cmd: echo "generated-password-123"
            setVariables:
              - name: DATABASE_PASSWORD
```

### After (Package Values)

```yaml
# zarf.yaml
values:
  files:
    - package-values.yaml

components:
  - name: app
    actions:
      onDeploy:
        after:
          - cmd: echo "user is {{ .Values.database.username }}"
            template: true  # Required for Go templates
          - cmd: echo "generated-password-123"
            setValues:
              - name: database.password
```

```yaml
# package-values.yaml
database:
  username: "postgres"
```
